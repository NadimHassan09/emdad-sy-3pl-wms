1. Executive Summary
The current dual-frontend WMS suffers from visual fragmentation, inconsistent interaction patterns, and missing enterprise SaaS affordances rather than structural defects. The backend integration, realtime Socket.IO plumbing, TanStack Query caching, and workflow logic are production-grade. The refactor mission is to modernize the perception and operability of the system without touching API contracts, route definitions, or business rules.
Core thesis: Because ~70% of the UI surface is data-dense tables, filters, and cards, modernizing the DataTable ecosystem first will yield disproportionate visual and UX impact across both applications. We proceed layer-by-layer (Design Tokens → Shared Primitives → Layout Shell → Table System → Workflow Visualization → Polish) rather than page-by-page to avoid partial-state inconsistency during rollout.

2.2 Admin Dashboard UX Problems
Navigation & Information Architecture
No breadcrumbs — Deep ledger drill-down (/inventory/ledger/:referenceType/:referenceId → line entry) relies on inline back links. Warehouse operators lose spatial context when jumping between orders, tasks, and ledger references.
WarehousesPage orphan — Full CRUD UI exists but is unregistered in router.tsx. Dead code or hidden capability creates maintenance liability.
Users route nav-gated but not route-gated — Sidebar hides Users for non-ADMIN, yet router allows direct navigation. API 403 is the only guard.
Legacy redirects (/inbound, /outbound, /dashboard) — Useful for bookmarks but increase mental model complexity without a redirect notice pattern.
Warehouse & Tenant Model
Single-warehouse bias — VITE_DEFAULT_WAREHOUSE_ID and useDefaultWarehouseId() assume single-warehouse UI. Multi-warehouse operators lack a global warehouse switcher in the shell.
Realtime dev-gated — VITE_MOCK_COMPANY_ID required for Socket.IO connection. Production must either use the same env hack or socket never connects, indicating a configuration-not-UI problem that surfaces as "stale data."
X-Company-Id header injection — Dev tenant injection pattern leaks into production mental model; no UI-level tenant scoping or indicator.
Task Execution
Monolithic TaskExecutionView (~2,500 lines) — High cognitive load for onboarding, testing, and bug isolation. Task-type-specific UI sections (receive, putaway, pick, pack, dispatch) are colocated rather than composed.
Task-only vs legacy receive split — VITE_TASK_ONLY_FLOWS flag bifurcates inbound confirm behavior. Operators may encounter different UX paths for the same business object depending on deployment config.
Exit blocker surprise — useBlocker prevents navigation with unsaved draft progress, but no persistent "unsaved changes" banner precedes the browser-level confirmation, creating jarring interruptions.
Lists & Filters
Client-side pagination in DataTable — Large datasets load entirely into memory before pagination. This creates a performance ceiling on product catalogs, inventory ledgers, and order histories.
Apply vs draft filter pattern — Consistent across pages but requires an extra click. Live/debounced filtering is standard SaaS expectation; current pattern slows operator scanning.
Visual Design
Triple primary color collision — Sidebar active states use #1a7a44, buttons use emerald-600, and Tailwind config defines primary as blue #2563eb. No semantic hierarchy exists (e.g., "action" vs "navigation" vs "accent").
No dark mode — Despite CSS variable architecture suggesting extensibility, no prefers-color-scheme or toggle implementation exists.
Mixed styling systems — Tailwind utilities + shared globals.css semantic classes + inline style={{}} on detail pages (client portal headers, admin metadata grids). This trifurcation makes global density adjustments impossible.
Accessibility
Table row keyboard patterns inconsistent — Client portal adds role="link" with Enter/Space handlers; admin uses onRowClick on <tr> without equivalent keyboard support.
Modal focus trap undocumented — No verified focus management in Modal/ConfirmModal component code.
Language attribute mismatch — Admin sets lang attribute when AR selected, but systematic icon mirroring for RTL not reviewed.
2.3 Client Portal UX Problems
Feature Gaps (Intentional but Visually Unpolished)
Read-only order surface — No creation, confirmation, or cancellation. By design, but the "view-only" state lacks visual affordance (e.g., read-only badges, disabled-action tooltips) that would clarify the portal's role to users.
No exports — PDF/CSV absent from all lists. Standard enterprise expectation for audit and reporting.
No operational dashboard — Welcome page shows profile metadata only. No KPI cards, open order counts, or stock alerts.
Stale Data
Realtime scope too narrow — All six socket events invalidate only ['client', 'stock']. Inbound/outbound order lists and detail pages do not auto-refresh when warehouse operators confirm or ship orders.
Visual Parity
Status as plain text on lists — Detail pages use .badge-* classes; lists show raw status strings. Inconsistent scanability forces users to read rather than recognize.
Inline styles on detail pages — Flex headers and back links use style={{}}, diverging from shared semantic classes and creating spacing drift.
Mobile & Density
Horizontal scroll only — table-wrap scroll containers on client; admin uses overflow-x-auto. Neither implements column prioritization, sticky first-column, or card-flip patterns for dense tables on small viewports.
Task execution density — Forms in TaskExecutionView may overflow on mobile without responsive field reordering.

4. Progressive Refactor Strategy
Core Principle: Layer-by-layer horizontal refactor. Each layer touches all pages simultaneously via shared components, avoiding the "half-old half-new" page problem.
Step 0 — Design Token Foundation (Week 1)
Goal: Establish the single source of truth before any UI changes.
Actions:
Expand shared/design-system/globals.css with full token set (colors, spacing, typography, shadows, radius)
Update tailwind.preset.cjs to map all utilities to CSS variables
Create shared/design-system/tokens.ts for JS/TS runtime access (badge colors, animation durations)
Zero UI changes yet — only map existing values to new tokens
Validation: Both apps compile. Visual output identical to before (tokens match current colors).
Step 1 — Shared Primitives & DataTable System (Weeks 2–4)
Goal: Unify the 70% of UI that is tables, filters, pagination, cards, and headers.
Actions:
1.1 Shared Component Package
Create shared/ui-components/ (or publish as internal package) containing:
Button (all variants, loading state, sizes)
TextField, SelectField, Combobox (with async search)
Card, CardHeader, CardContent
Badge (unified status mapping)
Modal, ConfirmModal, Drawer
ToastProvider, useToast
PageHeader (with breadcrumb slot)
FilterBar, FilterChip, FilterActions
EmptyState, Skeleton (table, card, text variants)
Breadcrumb (new)
Admin/Client Migration:
Admin: Replace Button, TextField, Modal, ToastProvider, PageHeader, StatusBadge
Client: Replace all inline styles and hand-rolled patterns with shared primitives
1.2 Unified DataTable Architecture
New DataTable Specification:
Server-side pagination first (unify 20/25 divergence to 25)
Density toggle (comfortable/compact/dense)
Column visibility toggle (for responsive overflow)
Sticky header + sticky first column (mobile optimization)
Row selection (checkbox) for bulk actions
Sort indicators (click header)
Empty state and skeleton integration
Keyboard navigation (arrow keys, enter to open)
Implementation:
Refactor admin DataTable to new spec
Replace client hand-rolled tables with shared DataTable
Update all list pages simultaneously: ProductsPage, InventoryPage, InboundListPage, OutboundListPage, TasksListPage, ClientProductsPage, ClientStockPage, ClientInboundOrdersPage, ClientOutboundOrdersPage
1.3 Unified Pagination
Component: Pagination (shared)
Server-side integration: Update API hooks to pass limit/offset (or page/pageSize)
Client-side fallback: Only for small datasets (<100 rows) where server pagination unavailable

6.4 Table Responsiveness Strategy
Problem: Horizontal scroll on dense tables is unavoidable for enterprise data, but can be managed.
Solutions:
Column Priority System: Mark columns as primary (always visible), secondary (hidden on tablet), tertiary (hidden on mobile). Primary = ID, Status, Key quantity. Tertiary = timestamps, metadata.
Sticky First Column: Order # / SKU stays visible while scrolling horizontally.
Card Flip (Mobile extreme): On <480px, optional card view instead of table (future enhancement, not required for refactor).
Density Auto-Reduce: Switch to dense mode automatically on mobile.
